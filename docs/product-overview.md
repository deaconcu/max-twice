# TwiceMax / 两清 - 产品概览

> 给以后进入项目的 AI / 协作者的一份"不看代码就能读懂产品"的摘要。
> 本文件只描述 **产品是什么、为什么这么设计**，不描述代码实现。

---

## 1. 一句话定位

> **What should it take to learn a bite of knowledge? Twice, Max.**

学一小块知识，最多过两遍。

- 域名：`twicemax.com`
- 英文名：TwiceMax
- 中文名：两清（"两次清楚" 的缩写，取 "看两遍就清楚了" 之意）
- 点赞按钮文案："Twicemax" —— 同时也是网站核心排序信号

"Twice, Max" 是地道英语口语结构，类似 "twice, tops"（最多两次）。不是生造词。

---

## 2. 要解决的问题

用户痛点："学了记不住、学了用不上。"

现有学习/知识平台的问题：
- 知乎类：长篇大论、抒情多于执行、回答之间无结构
- 得到/B 站课程：线性、被动、脱离用户具体角色需求
- 文档/教材：脱离场景，不知道学完能用在哪里

**TwiceMax 想做的是：** 让"两遍就能读懂"的内容自然排到前面，用结构化的节点 + 角色导向把"学 → 用" 的链路缩短。

**产品定位**：学习型产品（非消费型）。用户价值不是单一爆点内容，而是**一条完整动线**：
角色 → Roadmap → 课程 → 节点 → 文章。任何一环断档整条路就废，冷启动必须保证这条链路有若干条完整可走的样板。

## 3. 核心设计（4 个支点）

### 3.1 角色驱动（Role-driven）

不是职业驱动，不是学科驱动，是**角色**驱动。

- 角色包含职业角色（前端工程师、产品经理），也包含**生活角色**（新手父母、糖尿病患者、想学吉他的人）
- 角色是全局对象，不属于某个用户
- 用户通过选择/跟随角色来组织自己的学习

**角色下可以提交"学习路径"**：
- 学习路径是**由课程和节点组成的 DAG**（有向无环图），展示为 Roadmap
- 同一个角色下可以存在多条学习路径（面试向、项目向、兴趣向……）
- 学习路径由社区用户提交，官方或高权重用户也可产出
- 学习路径是角色 → 具体学习内容（课程 / 节点）之间的桥梁

**Roadmap 展示与编辑**：
- 使用 **Vue Flow** 做编辑器和展示
- 编辑：用户搜索课程/节点添加到画布，拉线建立前后关系，提交后进入审核
- 展示：展示页就是同一张图，节点需区分已学/未学状态
- 移动端需退化为列表视图（Vue Flow 在小屏不可用）

**为什么**：人的学习动机从来不是 "我想学线性代数"，而是 "我要做 X 事，需要懂 Y"。角色定义目标，学习路径告诉用户"要达到这个角色，按什么顺序学什么"。

### 3.2 节点（Node）是顶级资产

Node 是这个项目的**顶级资产**，比课程、文章、用户都更基础。

- 每个节点代表一个**原子知识点**（例：「梯度下降」、「React 的 useEffect 依赖数组」）
- 节点有**定义**（description），这个定义规定了"在这个节点下发表的文章的解释范围"
- 课程、Index、文章都围绕节点组织
- 同一个节点可以被多个课程、多个 Index 引用

**为什么**：知识点是最稳定的颗粒。课程会过时，文章会被淘汰，但"梯度下降是什么"这个节点不会消失。把节点作为顶级资产，意味着用户的学习痕迹、内容的价值都沉淀在节点上，不会因为某个课程下架就丢失。

### 3.3 Index Post（节点目录）是用户可选对象

Post 有两种类型：
- **普通文章**：针对某个节点的一篇解释 / 实战 / 笔记
- **Index Post**：**某个节点下的目录**，把该节点的子节点 / 相关节点按某种顺序组织起来

关键规则：
- Index post **挂在单个节点下**（不是课程级、不是主题级）
- **同一个节点下可以有多个 Index post**，由用户提交
- 用户在学习某个节点时，可以**自由选择**其中一个 Index post 作为自己在这个节点下的学习路径
- 用户在课程里遇到的每个节点都做一次这种选择，**多个被选中的 Index post 拼起来，就组成了这个用户在这门课程上的专属目录**
- **课程目录**展示在课程页左侧栏，交互同普通文档站（MDN / Vue 文档），区别是可根据当前阅读节点动态调整、支持多目录切换

**为什么**：同一个节点（如「梯度下降」），不同人的最佳展开顺序不一样——数学背景强的人想直接看证明，工程背景强的人想先看代码实现。把"节点展开顺序"这件事从"课程作者的特权"解放成"社区可贡献的资产"，就让每个用户都能在节点粒度上拼出最适合自己的学习路径，而不是被单一作者的叙述顺序绑架。

#### 3.3.1 Index Post 的提交与节点治理

**提交流程**：
1. 用户创建 Index post 时，需要选择子节点 / 相关节点填入目录
2. 选择节点时系统提供**向量搜索**（基于节点的名称 + 描述），避免用户创建重复节点
3. 用户可以**引用**已有节点，或**创建**新节点
4. 提交后进入**管理员审核**
5. 审核要点：如果 Index post 中含有与现有节点重复或高度相似的**新节点**，该 Index post 会被**拒绝**（用户需改为引用已有节点）

**后台治理**：
- 每日定时任务扫描全库节点的向量相似度
- 输出"疑似重复节点"列表供管理员决策
- 管理员可执行**节点合并**操作（将相似/重复节点归一）

**为什么这样设计**：
- 节点是顶级资产，如果节点集无序膨胀，整个系统的沉淀价值会被稀释
- 把"节点治理"放在 Index post 提交这一步拦截，是在**内容入口**处守门，成本最低
- 向量搜索 + 人工审核 + 定期扫描，是"防重复"的三道闸：用户侧前置建议、提交时强校验、后台兜底清理

### 3.4 Twicemax 投票

每篇文章可以被 "Twicemax"（点赞按钮名即排序信号）。

投票只有一个维度：**两遍读懂**（Twice, Max —— 最多看两遍就懂了）。

排序时把**两遍读懂**得票高的文章顶到前面。

**为什么**：传统点赞只有"喜欢/不喜欢"一个模糊维度，会让抒情长文、情绪化表达胜出。TwiceMax 明确投的是"这东西是不是真的让我快速看懂了"，内容生态会向**短、清、密度高**的方向倾斜。

这也是产品名 / 域名 / slogan / 按钮文案四位一体的来源——网站排序逻辑就是 slogan 本身。

## 4. 认证与账户

- 主登录：**邮箱 OTP 验证码**（不存在则自动建号，无密码）
- 辅助登录：邮箱 + 密码
- CAPTCHA：Cloudflare Turnstile，默认隐藏，失败次数超阈值（IP 维度）才强制校验
- 设置密码：登录后可通过 `/settings/password` 用 OTP 验证后设置（TODO，规划中）
- Session：Sa-Token
- 失败频控：`AuthRiskDomainService`，3 次失败/15 分钟 TTL，超阈值要求 Turnstile

认证改造历史：原来有 register / verify-email / password-reset 等一堆端点，已简化为 4 个：
- `/v1/auth/login/send-code`
- `/v1/auth/login/verify-code`
- `/v1/auth/login/resend-code`
- `/v1/auth/login/password`

AuthService 里保留了 register / password-reset 等方法（为将来复用），但当前 Controller 不暴露。

## 5. 页面设计风格

- **落地页**（未登录用户 = 登录页左侧引导区）：
  - 六屏手绘风 SVG（feTurbulence + feDisplacementMap 做手绘抖动）
  - 第一屏就是核心 slogan "Twice, Max"
- **HomePage**（已登录）：
  - 四步引导：**角色 → 路线 → 课程 → 复习**
  - 欢迎语后面会有设置密码的入口图标（未设置密码时）
- 组件库：Vuetify
- 国际化：vue-i18n，当前中/英
- 主题：手绘、温暖、克制，避免大厂科技风

## 6. 技术架构（极简版）

### 后端（Spring Boot 多模块）

```
backend/
├── shared/          # 通用 DTO / 异常 / 配置
├── user/            # 用户域（profile / auth / risk）
├── content/         # 内容域（course / node / post / index）
├── interaction/     # 互动（follow / twicemax 投票 / bookmark）
├── learning/        # 学习进度（enrollment / progress）
├── memory/          # 记忆卡片（SRS）
├── analytics/       # 统计
├── infrastructure/  # 基础设施（captcha / email / redis / datasource）
├── application/     # 应用服务层（编排）
├── web/             # REST API 层
└── tools/           # 管理工具
```

分层：**Controller → AppService（编排）→ DomainService → DataService**

### 前端

```
web/                 # 面向用户
admin-web/           # 面向管理员
```

Vue 3 + TypeScript + Vuetify + Pinia + vue-i18n。

## 7. 已知的架构张力 / 待解决问题

这些不是 bug，是产品模型本身带来的设计挑战，**在决定新功能时要考虑这些**：

1. **投票语境漂移**：同一节点被不同课程引用时，"两遍读懂"的标准会漂（在入门课里读懂的文章，在进阶课里可能不够）。目前没有完美解法。
2. **Index post 排序/推荐**：同一节点下会有多个 Index post，如何给学习者推荐合适的？（治理侧的重复节点问题已设计：向量搜索 + 审核 + 每日扫描合并，见 3.3.1）
3. **节点合并的历史影响**：合并节点时，被合并节点下的文章、投票、Index post、用户学习记录如何迁移和保留？机制需要细化。
4. **冷启动**：UGC 双边问题。当前思路是 AI 生成基准线内容（非顶部，作为"默认有货"）。
5. **模型复杂度**：角色 / 学习路径 / 节点 / Index post / 课程 / Twicemax 六个概念对新用户的理解门槛需要持续打磨入口引导。
