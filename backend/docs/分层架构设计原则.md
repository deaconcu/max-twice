# 分层架构设计原则

## 概述

本项目采用标准的三层架构模式：Controller → Service → Repository，每层有明确的职责边界，确保代码的可维护性、可测试性和可扩展性。

## API 响应设计原则

### 业务状态码与 HTTP 状态码分离

**核心理念**：HTTP 状态码表示传输层状态，业务状态码表示业务层结果。请求到达业务层后，统一返回 200 状态码，通过业务状态码区分具体的业务结果。

#### 1. 统一响应格式
```java
{
  "code": 0,           // 业务状态码：0=成功，非0=各种业务错误
  "message": "success", // 状态描述
  "data": { ... },     // 业务数据
  "timestamp": 1640995200000,
  "path": "/api/v1/users/123"
}
```

#### 2. 业务状态码设计规范
```java
public class ApiResponse<T> {
    // ==================== 成功码 ====================
    public static final int SUCCESS = 0;
    
    // ==================== 通用错误 10xx ====================
    public static final int PARAM_ERROR = 1001;     // 参数错误
    public static final int UNAUTHORIZED = 1002;    // 未授权
    public static final int FORBIDDEN = 1003;       // 禁止访问
    public static final int NOT_FOUND = 1004;       // 资源不存在
    public static final int RATE_LIMIT = 1005;      // 限流
    public static final int BUSINESS_ERROR = 1006;  // 通用业务错误
    
    // ==================== 用户模块错误 11xx ====================
    public static final int USER_NOT_FOUND = 1101;
    public static final int USER_ALREADY_EXISTS = 1102;
    public static final int USER_PASSWORD_ERROR = 1103;
    public static final int USER_EMAIL_EXISTS = 1104;
    public static final int USER_DISABLED = 1105;
    
    // ==================== 课程模块错误 12xx ====================
    public static final int COURSE_NOT_FOUND = 1201;
    public static final int COURSE_ACCESS_DENIED = 1202;
    public static final int COURSE_ALREADY_EXISTS = 1203;
    public static final int COURSE_NOT_APPROVED = 1204;
    
    // ==================== 系统错误 9xxx ====================
    public static final int SYSTEM_ERROR = 9999;
    public static final int DATABASE_ERROR = 9001;
    public static final int REDIS_ERROR = 9002;
    public static final int FILE_UPLOAD_ERROR = 9003;
}
```

#### 3. 错误码分段规则
- **0**：成功
- **1xxx**：客户端错误
  - **10xx**：通用错误（参数、权限等）
  - **11xx**：用户模块错误（每个模块99个错误码）
  - **12xx**：课程模块错误
  - **13xx**：评论模块错误
  - **14xx**：职业模块错误
  - **15xx**：路线图模块错误
- **9xxx**：系统错误

#### 4. HTTP 状态码使用原则

##### ✅ 统一返回 200 - 业务处理成功
```java
// 所有业务处理成功的情况都返回 200
@PostMapping("/users")
public ApiResponse<User> createUser(@RequestBody CreateUserRequest request) {
    if (userService.existsByEmail(request.getEmail())) {
        return ApiResponse.error(USER_EMAIL_EXISTS, "邮箱已存在");  // 200 + 业务错误码
    }
    
    User user = userService.create(request);
    return ApiResponse.success(user);  // 200 + 成功
}

// 前端统一处理
fetch('/api/users', { method: 'POST', body: userData })
  .then(response => response.json())  // 总是 200，可以直接解析
  .then(result => {
    if (result.code === 0) {
      // 成功处理
    } else {
      // 根据 result.code 处理具体业务错误
      switch(result.code) {
        case 1102: // USER_EMAIL_EXISTS
          showError("邮箱已存在");
          break;
        case 1001: // PARAM_ERROR
          showError("参数错误");
          break;
      }
    }
  });
```

##### ✅ 非 200 状态码 - 基础设施问题
```java
// 只在这些情况下使用非 200 状态码
@ExceptionHandler(Exception.class)

}
```

#### 5. 对比传统做法的优势

##### ❌ 传统做法 - HTTP 状态码混用
```java
// 问题：HTTP 状态码被用于表示业务状态
@PostMapping("/users")
public ResponseEntity<User> createUser(@RequestBody CreateUserRequest request) {
    if (userService.existsByEmail(request.getEmail())) {
        return ResponseEntity.status(409).build();  // 409 Conflict
    }
    
    User user = userService.create(request);
    return ResponseEntity.ok(user);  // 200 OK
}

// 前端需要复杂的状态码判断
fetch('/api/users')
  .then(response => {
    if (response.status === 200) {
      return response.json();
    } else if (response.status === 409) {
      throw new Error("邮箱已存在");
    } else if (response.status === 400) {
      throw new Error("参数错误");
    } else if (response.status === 500) {
      throw new Error("服务器错误");
    }
  })
  .catch(error => {
    // 错误处理复杂，需要区分网络错误和业务错误
  });
```

##### ✅ 推荐做法 - 状态码分离
```java
// HTTP 状态码只表示传输层状态，业务状态通过 code 字段表示
@PostMapping("/users")
public ApiResponse<User> createUser(@RequestBody CreateUserRequest request) {
    if (userService.existsByEmail(request.getEmail())) {
        return ApiResponse.error(USER_EMAIL_EXISTS, "邮箱已存在");  // 200 + 1102
    }
    
    User user = userService.create(request);
    return ApiResponse.success(user);  // 200 + 0
}

// 前端处理简化
fetch('/api/users')
  .then(response => response.json())  // 总是可以解析，因为总是 200
  .then(result => {
    if (result.code === 0) {
      // 成功
    } else {
      // 业务错误，根据 code 显示具体错误信息
      showError(result.message);
    }
  })
  .catch(error => {
    // 只需要处理网络错误
    showError("网络异常，请检查网络连接");
  });
```

#### 6. 实现示例

##### Controller 层统一返回格式
```java
@RestController
public class UserController {
    
    // ✅ 正确：统一返回 ApiResponse，HTTP 状态码总是 200
    @PostMapping("/users")
    public ApiResponse<User> createUser(@RequestBody CreateUserRequest request) {
        try {
            User user = userService.createUser(request);
            return ApiResponse.success(user);  // code: 0
        } catch (EmailExistsException e) {
            return ApiResponse.error(USER_EMAIL_EXISTS, "邮箱已存在");  // code: 1102
        } catch (InvalidParameterException e) {
            return ApiResponse.error(PARAM_ERROR, e.getMessage());  // code: 1001
        }
    }
    
    // ❌ 错误：混用 HTTP 状态码表示业务状态
    @PostMapping("/users/old")
    public ResponseEntity<User> createUserOld(@RequestBody CreateUserRequest request) {
        if (userService.existsByEmail(request.getEmail())) {
            return ResponseEntity.status(409).build();  // 业务错误用了 HTTP 状态码
        }
        User user = userService.createUser(request);
        return ResponseEntity.ok(user);
    }
}
```

##### 全局异常处理
```java
@RestControllerAdvice
public class GlobalExceptionHandler {
    
    // 业务异常 - 返回 200 + 业务错误码
    @ExceptionHandler(BusinessException.class)
    public ApiResponse<Object> handleBusinessException(BusinessException e) {
        return ApiResponse.error(e.getCode(), e.getMessage());
    }
    
    // 参数验证异常 - 返回 200 + 参数错误码
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ApiResponse<Object> handleValidationException(MethodArgumentNotValidException e) {
        String message = e.getBindingResult().getAllErrors().get(0).getDefaultMessage();
        return ApiResponse.error(PARAM_ERROR, message);
    }
    
    // 系统异常 - 返回 500 + 系统错误码
    @ExceptionHandler(Exception.class)
    public ResponseEntity<ApiResponse<Object>> handleSystemException(Exception e) {
        log.error("系统异常", e);
        return ResponseEntity.status(500)
            .body(ApiResponse.error(SYSTEM_ERROR, "系统繁忙，请稍后重试"));
    }
}
```

#### 7. 好处总结

1. **前端处理统一**：所有业务接口都是 200 + JSON，解析逻辑一致
2. **错误码体系化**：业务错误码有明确的模块划分和扩展规则
3. **调试更简单**：HTTP 5xx 表示服务故障，200 + code ≠ 0 表示业务错误
4. **API 网关友好**：网关可以统一处理，所有 200 都是业务成功到达
5. **监控更清晰**：可以分别监控传输层问题和业务层问题
6. **国际化友好**：错误码可以映射到不同语言的错误信息

```java
@Service
public class UserService {

    public UserDTO validateLogin(String email, String password) {
        UserDO userDO = userMapper.getByEmail(email);
        if (userDO == null) {
```

```java
@Service
public class UserService {
    
    private final MessageService messageService;
    private final LearningProgressService learningProgressService;
    
    @Transactional
    public void follow(Long followerId, Long followeeId) {
        // 业务规则检查
        if (followerId.equals(followeeId)) {
            throw ErrorCode.CANNOT_FOLLOW_SELF.exception();
        }
        
        UserDO followee = userMapper.getById(followeeId);
        if (followee == null) {
            throw ErrorCode.USER_NOT_FOUND.exception();

        follow.setFollowerId(followerId);
        follow.setFolloweeId(followeeId);
        followMapper.insert(follow);
        
        // 发送关注通知（跨服务调用）
        UserDO follower = userMapper.getById(followerId);
        messageService.createFollowMessage(followeeId, follower.getId());
    }
}
```

## 分层交互原则

### 1. 依赖方向
```
Controller → Service → Repository
```
- 上层可以依赖下层，下层不能依赖上层
- Service 不能直接调用 Controller
- Repository 不能调用 Service

### 2. 数据传递
```java
// ✅ 正确：Controller 提取认证信息，传递给 Service
@PostMapping("/courses")
public ApiResponse<Course> createCourse(@RequestBody CreateCourseRequest request) {

    Course course = courseService.createCourse(request, userId);  // 传递给 Service
    return ApiResponse.success(course);

@Service

        Long userId = StpUtil.getLoginIdAsLong();  // ❌ Service 不应该知道认证框架
        // ...
    }
}
```

### 3. 异常处理层次

## 错误处理最佳实践

### ErrorCode 统一错误码管理

项目使用 `ErrorCode` 枚举统一管理所有错误码，确保错误处理的一致性和可维护性。

#### 1. ErrorCode 设计原则
```java
public enum ErrorCode {
    // 用户认证相关 A0xxx
    USER_NOT_LOGIN("A0001", "用户未登录"),
    PERMISSION_DENIED("A0002", "权限不足"),
    USER_NOT_FOUND("A0003", "user.not.found"),
    
    // 普通业务相关 B00xx
    NOT_SUPPORTED("B0001", "不支持的操作类型"),
    INVALID_PARAMETER("B0002", "参数异常"),
    
    // 课程相关 B01xx
    COURSE_NOT_FOUND("B0101", "课程不存在"),
    
    // 内容管理相关 B06xx
    CONTENTS_COURSE_NOT_FOUND("B0601", "课程不存在"),
    CONTENTS_POST_NOT_FOUND("B0602", "帖子不存在"),
    CONTENTS_USER_TOC_NOT_FOUND("B0603", "用户目录不存在"),
    CONTENTS_TOC_INDEX_OUT_OF_BOUNDS("B0604", "目录索引超出范围"),
    CONTENTS_PINNED_ITEMS_LIMIT_EXCEEDED("B0605", "置顶帖子数量超限"),
    CONTENTS_INVALID_POST_TYPE("B0606", "无效的帖子类型"),
    CONTENTS_JSON_PROCESSING_ERROR("B0607", "JSON处理异常"),
```

```java
            // JSON 处理逻辑
        } catch (JsonProcessingException e) {
            throw ErrorCode.CONTENTS_JSON_PROCESSING_ERROR.exception(e);

    }
```

#### 4. 全局异常处理器
```java
@RestControllerAdvice
public class GlobalExceptionHandler {
    
    @ExceptionHandler(BusinessException.class)
    public ApiResponse<Object> handleBusinessException(BusinessException e) {
        log.warn("业务异常: {}", e.getMessage());
        return ApiResponse.error(e.getCode(), e.getMessage());
    }
    
    @ExceptionHandler(Exception.class)
    public ResponseEntity<ApiResponse<Object>> handleSystemException(Exception e) {
        log.error("系统异常", e);
        return ResponseEntity.status(500)
            .body(ApiResponse.error(ErrorCode.SYSTEM_ERROR.getCode(), "系统繁忙，请稍后重试"));
    }
}
```

#### 5. ErrorCode 使用规范

1. **命名规范**：`模块_具体错误`
   - `CONTENTS_COURSE_NOT_FOUND`：内容模块-课程不存在
   - `USER_NOT_FOUND`：用户模块-用户不存在

2. **分段规则**：
   - A0xxx：认证相关
   - B0xxx：业务相关（B01xx=课程，B02xx=路线图，B06xx=内容管理）
   - S0xxx：系统相关

3. **抛出方式**：
   ```java
   // 无参异常
   throw ErrorCode.COURSE_NOT_FOUND.exception();
   
   // 带原因异常
   throw ErrorCode.JSON_PROCESSING_ERROR.exception(cause);
   
   // 带自定义消息（国际化）
   throw ErrorCode.USER_NOT_FOUND.exception("用户不存在");
   ```

4. **禁止使用**：
   ```java
   // ❌ 禁止直接抛出 RuntimeException
   throw new RuntimeException("course is not exist");
   
   // ❌ 禁止使用字符串错误码
   throw new BusinessException("COURSE_NOT_FOUND", "课程不存在");
   ```

## 总结

遵循清晰的分层边界能够：

1. **提高可测试性**：每层可以独立测试
2. **增强可维护性**：修改某层不影响其他层
3. **提升可扩展性**：可以独立替换某层的实现
4. **改善代码复用**：Service 层可以被多种 Controller 调用
5. **简化调试**：问题定位更容易

**关键原则**：
- **Controller**：HTTP + 认证 + 参数验证
- **Service**：纯业务逻辑
- **Repository**：纯数据访问

每层只做自己职责范围内的事情，不越界、不跨层。
