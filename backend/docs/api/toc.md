# 课程目录管理接口文档 (TocController)

## 基本信息

- **Controller**: `TocController.java`
- **基础路径**: `/api/v1`
- **Rate Limit**: 60 requests/minute (per user)
- **前端 API**: `web/src/api/modules/page.ts` (部分功能)

## 概述

课程目录(Table of Contents, TOC)管理接口用于处理用户自定义的课程导航结构。系统采用基于哈希的目录版本管理机制，支持用户个性化调整课程目录结构。

**核心特性**:
- 用户可以自定义课程目录结构和节点顺序
- 使用哈希值去重和版本控制，节省存储空间
- 支持引用计数管理，优化存储效率
- 每个用户可以有独立的目录结构

## 领域模型说明

### CourseTocDO (课程目录版本)
用途：存储课程目录的具体结构，基于哈希去重

```java
{
  "hash": "a1b2c3d4...",           // 目录内容的 SHA 哈希值，作为唯一标识
  "toc": "{\"123\":{\"456\":{}}}",  // 目录结构的 JSON 字符串
  "refCount": 5,                    // 引用计数，有多少个用户使用该版本
  "createdAt": "2025-01-18 10:00:00"
}
```

**字段说明**：
- `hash` (String, 主键): 目录内容的 SHA 哈希值
  - 通过 `Utils.hashSHA(tocJson)` 生成
  - 相同内容的目录共享同一个哈希
  - 用于去重和版本控制
- `toc` (String): 目录结构的 JSON 字符串
  - 格式：`{"rootNodeId": {"childNodeId1": {...}, "childNodeId2": {...}}}`
  - 键为节点ID（字符串形式）
  - 值为子节点的嵌套结构
  - 空对象 `{}` 表示叶子节点
- `refCount` (Integer): 引用计数
  - 记录有多少个 `UserCourseTocDO` 引用了该版本
  - 用于垃圾回收和存储优化
- `createdAt` (LocalDateTime): 创建时间

**使用场景**：
- 存储课程目录的具体结构
- 通过哈希实现去重和共享
- 引用计数管理生命周期

---

### UserCourseTocDO (用户课程目录)
用途：记录用户对特定课程的目录结构

```java
{
  "id": 1,
  "userId": 83,
  "courseId": 123,
  "toc": "hash1,hash2,hash3",
  "updatedAt": "2025-01-18 10:00:00"
}
```

**字段说明**：
- `id` (Long): 记录ID
- `userId` (Long): 用户ID
- `courseId` (Long): 课程ID
- `toc` (String): 目录哈希数组（逗号分隔）
  - 格式：`"hash1,hash2,hash3,..."`
  - 每个哈希对应 `CourseTocDO` 表中的一个版本
  - 数组索引对应不同的目录层级或版本
  - 最多支持 9 个哈希值
- `updatedAt` (LocalDateTime): 最后更新时间

**数据关系**：
- 一个用户对一个课程有一条记录（userId + courseId 唯一）
- `toc` 字段中的每个哈希对应 `CourseTocDO.hash`
- 通过哈希数组支持多版本或多层级目录

**使用场景**：
- 记录用户自定义的课程目录结构
- 支持用户个性化的课程导航
- 跟踪用户的目录修改历史

---

### CourseTocDTO (课程目录响应)
用途：返回给前端的课程目录信息

```json
{
  "contents": "{\"123\":{\"456\":{}}}",
  "nodeInfos": {
    "123": {
      "id": 123,
      "name": "章节一",
      "completed": true,
      "progress": 0.8
    },
    "456": {
      "id": 456,
      "name": "小节一",
      "completed": false,
      "progress": 0.3
    }
  }
}
```

**字段说明**：
- `contents` (String): 目录结构的 JSON 字符串
  - 格式与 `CourseTocDO.toc` 相同
  - 键为节点ID，值为子节点结构
- `nodeInfos` (Map<Long, NodeWithProgressDTO>): 节点信息映射
  - 键: 节点ID
  - 值: 节点详情（包含名称、进度、完成状态等）
  - 动态填充：根据 `contents` 中的节点ID查询

**使用场景**：
- 返回给前端的课程目录数据
- 用于渲染课程导航树
- 显示节点的学习进度

## 接口列表

## 1. 更新用户课程目录

**接口路径**: `PUT /api/v1/users/current/courses/{courseId}/toc`

**是否需要登录**: 是 (`@SaCheckLogin`)

**路径参数**:
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| courseId | Long | 是 | 课程ID，必须大于0 |

**请求体** (使用 `@JsonParam`):
```json
{
  "indexArray": "1,2,0,3"
}
```

**请求参数说明**:
- `indexArray` (String, 必填): 索引数组（逗号分隔）
  - 格式：`"1,2,0,3,..."`
  - 长度限制：最多 9 个索引
  - 索引含义：
    - `0`: 使用默认目录结构（只包含根节点）
    - `正整数`: 使用现有目录数组中对应位置的哈希（从1开始）
    - `负整数`: 暂不支持（会报错）
  - 验证规则：
    - 不能为空
    - 每个索引的绝对值不能超过现有目录数组的长度
    - 索引必须是有效的整数

**业务逻辑说明**:

### 目录更新流程

1. **验证课程存在性**
   - 查询课程是否存在
   - 不存在返回 500 错误

2. **获取用户当前目录**
   - 查询 `UserCourseTocDO`
   - 解析 `toc` 字段为哈希数组

3. **生成默认目录**
   - 创建只包含根节点的目录结构：`{"rootNodeId": {}}`
   - 计算 SHA 哈希值
   - 如果不存在则插入到 `CourseTocDO` 表

4. **解析并验证索引数组**
   - 分割 `indexArray` 为整数数组
   - 验证长度不超过 9
   - 验证每个索引的有效性

5. **构建新的目录哈希数组**
   - 根据索引数组选择对应的哈希
   - `0` → 使用默认目录哈希
   - `正整数 n` → 使用 `tocHashes[n-1]`

6. **更新引用计数**
   - 统计使用默认目录的次数
   - 增加默认目录的 `refCount`

7. **保存新目录**
   - 将新哈希数组拼接为字符串
   - 更新 `UserCourseTocDO.toc`

### 示例场景

**场景1：重置部分目录为默认**
```
当前目录: "hashA,hashB,hashC"
索引数组: "1,0,3"
结果: "hashA,defaultHash,hashC"
```

**场景2：重新排列目录顺序**
```
当前目录: "hashA,hashB,hashC"
索引数组: "3,2,1"
结果: "hashC,hashB,hashA"
```

**场景3：使用全新的目录结构**
```
当前目录: "hashA,hashB"
索引数组: "0,0,0"
结果: "defaultHash,defaultHash,defaultHash"
```

**返回示例**:
```json
{
  "code": 200,
  "message": "操作成功",
  "data": "目录更新成功",
  "timestamp": 1766850806720
}
```

**错误场景**:
- 课程不存在 → 500
- 用户目录不存在 → 500
- 索引数组长度超过 9 → 500
- 索引值超出范围 → 500
- 索引值无法解析为整数 → 500

**前端调用**:
```typescript
// 暂无专门的 toc API，通常通过 page API 间接调用

// 可能的用法：
import apiClient from '@/api/client'

// 更新用户课程目录
const response = await apiClient.put(
  `/api/v1/users/current/courses/${courseId}/toc`,
  { indexArray: '1,2,0,3' }
)
```

**使用场景**:
- 用户自定义课程目录结构
- 调整章节顺序
- 重置某些章节为默认结构
- 课程个性化设置

## 2. 获取用户课程目录

**接口路径**: `GET /api/v1/users/current/courses/{courseId}/toc`

**是否需要登录**: 是 (`@SaCheckLogin`)

**路径参数**:
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| courseId | Long | 是 | 课程ID，必须大于0 |

**返回类型**: `String` (目录哈希数组)

**返回示例**:
```json
{
  "code": 200,
  "message": "操作成功",
  "data": "hash1,hash2,hash3",
  "timestamp": 1766850806720
}
```

**返回数据说明**:
- 返回用户当前的目录哈希数组（逗号分隔的字符串）
- 每个哈希对应 `CourseTocDO` 表中的一个目录版本
- 前端需要进一步查询具体的目录内容（通过其他接口）

**错误场景**:
- 课程不存在 → 500
- 用户目录不存在 → 500

**前端调用**:
```typescript
// 暂无专门的 toc API

// 可能的用法：

// 获取用户课程目录

  `/api/v1/users/current/courses/${courseId}/toc`

```

**使用场景**:
- 加载用户自定义的课程目录
- 获取当前目录版本
- 判断目录是否需要更新

## 错误码说明

| 错误码 | 说明 |
|--------|------|
| 200 | 成功 |
| 400 | 参数验证失败 |
| 401 | 未登录（需要登录才能操作目录） |
| 403 | 无权限 |
| 404 | 资源不存在（课程不存在） |
| 429 | 请求频率超限（60次/分钟） |
| 500 | 服务器内部错误（课程不存在、目录不存在、索引无效等） |

**常见参数验证错误**:
- `课程ID不能为空` - courseId 为 null
- `课程ID必须大于0` - courseId ≤ 0
- `索引数组不能为空` - indexArray 为 null 或空字符串

**业务错误**:
- 课程不存在 → `SYSTEM_ERROR` (500)
- 用户目录不存在 → `SYSTEM_ERROR` (500)
- 索引数组长度超过 9 → `SYSTEM_ERROR` (500)
- 索引值超出范围 → `SYSTEM_ERROR` (500)
- 索引值格式错误 → `SYSTEM_ERROR` (500)

## 数据结构说明

### 目录结构格式

**目录 JSON 格式**:
```json
{
  "123": {
    "456": {},
    "457": {
      "789": {},
      "790": {}
    }
  }
}
```

**结构说明**:
- 键为节点ID（字符串形式）
- 值为子节点的嵌套对象
- 空对象 `{}` 表示叶子节点（无子节点）

**示例解析**:
```
课程 (rootNodeId=123)
├─ 节点 456 (叶子节点)
└─ 节点 457
   ├─ 节点 789 (叶子节点)
   └─ 节点 790 (叶子节点)
```

### 目录哈希数组

**格式**: `"hash1,hash2,hash3,..."`

**含义**:
- 每个哈希对应一个目录版本
- 数组长度通常为 1-9
- 可以用于：
  - 多版本目录管理
  - 目录历史记录
  - A/B 测试不同的目录结构

**示例**:
```
toc = "abc123,def456,ghi789"

索引 1 → abc123 (对应某个目录结构)
索引 2 → def456 (对应另一个目录结构)
索引 3 → ghi789 (对应第三个目录结构)
```

## 业务流程说明

### 目录更新流程图

```
用户请求更新目录
  ↓
Controller 验证课程ID和索引数组
  ↓
查询课程和用户当前目录
  ↓
生成默认目录（如果需要）
  ├─ 创建 {"rootNodeId": {}} 结构
  ├─ 计算 SHA 哈希
  └─ 插入到 CourseTocDO（如不存在）
  ↓
解析索引数组并构建新哈希数组
  ├─ 索引 = 0 → 使用默认哈希
  ├─ 索引 > 0 → 使用现有哈希[索引-1]
  └─ 索引 < 0 → 抛出错误
  ↓
更新引用计数
  ├─ 统计使用默认哈希的次数
  └─ CourseTocDO.refCount += count
  ↓
保存新目录到 UserCourseTocDO
  ↓
返回成功响应
```

### 哈希去重机制

1. **生成哈希**
   - 将目录 JSON 字符串通过 SHA 算法生成哈希值
   - 相同内容的目录生成相同的哈希

2. **查询或插入**
   - 检查 `CourseTocDO` 表中是否存在该哈希
   - 不存在则插入新记录（refCount = 0）
   - 存在则复用现有记录

3. **引用计数管理**
   - 新增引用时 `refCount++`
   - 删除引用时 `refCount--`
   - `refCount = 0` 时可以清理（垃圾回收）

4. **优点**
   - 节省存储空间（相同目录只存一份）
   - 支持版本管理
   - 便于统计使用情况

## 注意事项

1. **登录要求**：
   - 所有目录接口都需要登录
   - 只能操作当前用户的目录

2. **索引数组限制**：
   - 最多 9 个索引
   - 索引从 1 开始（不是从 0）
   - 索引 0 表示使用默认目录

3. **默认目录**：
   - 只包含根节点：`{"rootNodeId": {}}`
   - 自动创建和管理
   - 多个用户可以共享同一个默认目录

4. **哈希机制**：
   - 使用 SHA 算法生成哈希
   - 相同内容的目录共享同一个哈希
   - 通过引用计数管理生命周期

5. **错误处理**：
   - 大部分业务错误返回 500（`SYSTEM_ERROR`）
   - 建议细化错误码，区分不同的业务错误

6. **性能考虑**：
   - 哈希去重减少存储空间
   - 引用计数支持垃圾回收
   - 适合大量用户共享相同目录结构的场景

7. **数据一致性**：
   - 需要确保 `UserCourseTocDO.toc` 中的哈希都存在于 `CourseTocDO` 表中
   - 删除 `CourseTocDO` 记录前需要检查引用计数

## 改进建议

### 1. 错误码细化
当前所有业务错误都返回 `SYSTEM_ERROR` (500)，建议细化为：
- `COURSE_NOT_FOUND` - 课程不存在
- `USER_TOC_NOT_FOUND` - 用户目录不存在
- `INVALID_INDEX_ARRAY_LENGTH` - 索引数组长度超限
- `INVALID_INDEX_VALUE` - 索引值超出范围
- `INVALID_INDEX_FORMAT` - 索引格式错误

### 2. 接口优化
- 返回完整的 `CourseTocDTO` 而不只是哈希字符串
- 支持批量获取多个课程的目录
- 支持目录历史查询和回滚

### 3. 性能优化
- 缓存常用的目录结构
- 批量查询节点信息
- 异步更新引用计数

### 4. 功能增强
- 支持目录模板（预设的目录结构）
- 支持目录版本管理和回滚
- 支持目录分享（用户间共享自定义目录）
- 支持目录对比和合并
