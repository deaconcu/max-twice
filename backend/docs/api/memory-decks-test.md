- 获取帖子创建者卡片组接口

## 1. 获取帖子公共卡片组接口测试

### 1.1 功能测试

#### 测试用例 1.1.1: 按分数排序获取卡片组列表
- **测试目的**: 验证能正确按分数降序获取卡片组
- **前置条件**:
  - 数据库中存在多个已发布的卡片组
  - 卡片组具有不同的分数
- **测试步骤**:
  1. 发起请求 GET /api/v1/memory/posts/{postId}/decks
  2. 设置参数 sortBy=score
  3. 不传递 lastScore 和 lastId
- **预期结果**:
  - 返回状态码 200
  - data.items 数组包含卡片组
  - 卡片组按 score 降序排列（最高分在前）
  - 固定返回 20 条数据
  - 每条卡片组包含完整的 DeckWithVoteDTO 结构

#### 测试用例 1.1.2: 按时间排序获取卡片组列表
- **测试目的**: 验证能正确按创建时间降序获取卡片组
- **前置条件**:
  - 数据库中存在多个已发布的卡片组
- **测试步骤**:
  1. 发起请求 GET /api/v1/memory/posts/{postId}/decks
  2. 设置参数 sortBy=createdAt
  3. 不传递 lastId
- **预期结果**:
  - 返回状态码 200
  - data.items 数组包含卡片组
  - 卡片组按 id 降序排列（最新的在前）
  - 固定返回 20 条数据

#### 测试用例 1.1.3: 只返回正常状态的卡片组
- **测试目的**: 验证只返回 state=1（已发布）的卡片组
- **前置条件**:
  - 数据库中存在不同状态的卡片组（待审核、已发布、已拒绝、已封禁）
- **测试步骤**:
  1. 发起请求 GET /api/v1/memory/posts/{postId}/decks
  2. 设置参数 sortBy=score
- **预期结果**:
  - 返回状态码 200
  - 所有返回的卡片组 state=1
  - 不包含待审核、已拒绝、已封禁的卡片组

#### 测试用例 1.1.4: 未登录用户访问
- **测试目的**: 验证未登录用户可以查看公共卡片组
- **前置条件**:
  - 数据库中存在已发布的卡片组
- **测试步骤**:
  1. 不携带 token 发起请求
  2. 设置参数 sortBy=score
- **预期结果**:
  - 返回状态码 200
  - 正常返回卡片组列表
  - hasUpvoted 字段为 false

#### 测试用例 1.1.5: 已登录用户查看点赞状态
- **测试目的**: 验证登录用户能看到自己的点赞状态

### 1.2 分页测试

#### 测试用例 1.2.1: 按分数分页 - 首次查询
- **测试目的**: 验证按分数排序时首次查询不传游标参数
- **前置条件**:
  - 数据库中存在超过 20 个卡片组
- **测试步骤**:
  1. 发起请求，sortBy=score
  2. 不传 lastScore 和 lastId
- **预期结果**:
  - 返回状态码 200
  - 返回前 20 个最高分的卡片组
  - data.hasMore=true
  - data.nextCursor 包含 lastScore 和 lastId

#### 测试用例 1.2.2: 按分数分页 - 使用游标
- **测试目的**: 验证使用 lastScore 和 lastId 进行分页

- **测试步骤**:
  1. 第一次请求：sortBy=score，不传游标
  2. 记录返回的 nextCursor.lastScore 和 nextCursor.lastId
  3. 第二次请求：传递上次的 lastScore 和 lastId
- **预期结果**:
  - 两次请求都返回状态码 200
  - 第二次返回的卡片组分数都小于等于 lastScore
  - 相同分数的卡片组 id 小于 lastId
  - 数据不重复

#### 测试用例 1.2.3: 按时间分页 - 首次查询
- **测试目的**: 验证按时间排序时首次查询不传游标参数

- **测试步骤**:
  1. 发起请求，sortBy=createdAt
  2. 不传 lastId
- **预期结果**:
  - 返回状态码 200
  - 返回最新的 20 个卡片组
  - data.hasMore=true
  - data.nextCursor 包含 lastId

#### 测试用例 1.2.4: 按时间分页 - 使用游标
- **测试目的**: 验证使用 lastId 进行时间分页
- **前置条件**:
  - 数据库中存在超过 20 个卡片组
- **测试步骤**:
  1. 第一次请求：sortBy=createdAt，不传 lastId

  3. 第二次请求：传递上次的 lastId
- **预期结果**:
  - 两次请求都返回状态码 200
  - 第二次返回的所有卡片组 id 小于 lastId

#### 测试用例 1.2.5: 最后一页数据
- **测试目的**: 验证查询到最后一页时的行为
- **前置条件**:

## 13. 错误恢复测试

### 13.1 数据库异常

#### 测试用例 13.1.1: 数据库连接断开
- **测试目的**: 验证数据库连接断开时的错误处理
- **测试步骤**:
  1. 模拟数据库连接断开
  2. 发起查询请求
- **预期结果**:
  - 返回友好的错误信息
  - 系统不崩溃
  - 记录错误日志

#### 测试用例 13.1.2: 数据库查询超时
- **测试目的**: 验证数据库查询超时时的处理
- **测试步骤**:
  1. 模拟数据库查询缓慢
  2. 发起查询请求
- **预期结果**:
  - 在超时时间内返回错误信息
  - 不会无限等待
  - 释放数据库连接

#### 测试用例 13.1.3: 事务回滚
- **测试目的**: 验证事务异常时的回滚机制
- **测试步骤**:
  1. 创建卡片组，在插入卡片时模拟异常
  2. 检查数据库状态
- **预期结果**:
  - 事务完全回滚
  - 卡片组和卡片都未创建
  - 数据一致性保持

### 13.2 缓存异常

#### 测试用例 13.2.1: Redis 不可用
- **测试目的**: 验证 Redis 不可用时系统的降级处理
- **前置条件**: 系统使用 Redis 缓存
- **测试步骤**:
  1. 停止 Redis 服务
  2. 发起查询请求
- **预期结果**:
  - 系统降级到直接查询数据库
  - 接口正常返回数据
  - 记录缓存异常日志

#### 测试用例 13.2.2: 缓存数据不一致
- **测试目的**: 验证缓存与数据库数据不一致时的处理
- **测试步骤**:
  1. 修改数据库中的卡片组数据
  2. 缓存中保留旧数据
  3. 查询该卡片组
- **预期结果**:
  - 根据缓存策略返回数据
  - 缓存失效后返回最新数据
  - 或使用缓存穿透策略

### 13.3 服务异常

#### 测试用例 13.3.1: 内存溢出处理
- **测试目的**: 验证内存不足时的处理
- **测试步骤**:
  1. 模拟内存不足场景
  2. 发起大量请求
- **预期结果**:
  - 系统优雅降级
  - 返回服务不可用错误
  - 不会整体崩溃

#### 测试用例 13.3.2: CPU 占用过高
- **测试目的**: 验证CPU占用过高时的系统行为
- **测试步骤**:
  1. 模拟CPU密集型操作
  2. 发起请求
- **预期结果**:
  - 响应时间可能增加但仍能响应
  - 系统保持可用
  - 触发告警机制

| 错误码 | 说明 | 处理建议 |
|--------|------|---------|

| 1002 | 参数异常 | 检查请求参数是否正确 |
| 1006 | 权限不足 | 确认当前用户是否有操作权限 |
| 1101 | 用户未登录 | 重新登录获取 token |
| 1116 | 用户不存在 | 确认用户ID是否有效 |
| 2102 | 帖子不存在 | 确认帖子ID是否有效 |
| 2201 | 卡片组不存在 | 确认卡片组ID是否有效 |
| 2206 | 卡片组已存在于记忆库中 | 不能重复添加 |
| 2301 | 访问过于频繁 | 降低请求频率 |
| 9005 | AI服务请求失败 | 稍后重试或联系管理员 |

### B. 卡片组状态汇总

| state | 名称 | 说明 | 可见性 |
|-------|------|------|--------|
| 0 | pending | 待审核 | 仅创建者和管理员可见 |
| 1 | normal | 已发布 | 所有用户可见 |
| 2 | rejected | 已拒绝 | 仅创建者和管理员可见 |
| 3 | banned | 已封禁 | 仅管理员可见 |

| createdAt | 按时间排序 | ID降序（最新的在前） | lastId |
