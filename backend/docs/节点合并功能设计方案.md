# 节点合并功能设计方案

## 背景

节点是系统中最重要的知识单元概念。随着用户创建内容的增多，不可避免会出现重复或相似的节点（类似知乎的重复问题）。需要提供节点合并功能来维护知识体系的一致性。

## 业务场景

- 多个用户针对同一知识点创建了不同的节点（如："Java基础"、"Java入门"、"Java基础知识"）
- 这些节点的描述范围高度重合
- 需要将它们合并为一个节点，避免知识碎片化

## 核心方案

### 合并操作内容

#### 1. 需要迁移的数据

**1.1 文章迁移**
- 表：`post`
- 操作：批量更新 `node_id` 字段（源节点ID → 目标节点ID）
- SQL: `UPDATE post SET node_id = ? WHERE node_id = ?`

**1.2 记忆卡片组迁移**
- 表：`memory_card_deck`
- 操作：批量更新 `node_id` 字段
- SQL: `UPDATE memory_card_deck SET node_id = ? WHERE node_id = ?`

**1.3 SRS学习记录迁移**
- 表：`user_card_srs`
- 操作：批量更新 `node_id` 字段
- SQL: `UPDATE user_card_srs SET node_id = ? WHERE node_id = ?`
- **待确认问题1**：是否会出现冲突（同一用户在两个节点下学习同一卡片）？如何处理？

**1.4 课程目录更新**
- 表：`course_toc`
- 操作：解析 `toc` JSON字段，替换其中的节点ID
- 处理：
  - 将JSON中所有源节点ID替换为目标节点ID
  - 去重：如果替换后同一位置出现重复节点，只保留一个
- **待确认问题2**：去重时保留第一个还是最后一个？

**1.5 用户自定义目录更新**
- 表：`user_course_toc`
- 操作：解析 `toc` JSON字段，替换其中的节点ID
- 处理：同 course_toc

#### 2. 源节点状态处理

**节点表字段调整**
- 添加字段：
  ```sql
  ALTER TABLE node ADD COLUMN merged_to_node_id BIGINT DEFAULT NULL;
  ALTER TABLE node ADD COLUMN merged_at DATETIME DEFAULT NULL;
  ```
- 状态值：需要定义一个新的 state 值表示"已合并"（如：state=5）

**源节点标记**
- 设置 `state` = 已合并状态
- 设置 `merged_to_node_id` = 目标节点ID
- 设置 `merged_at` = 当前时间

**访问控制**
- 搜索节点时：过滤掉已合并状态的节点
- 访问节点时：自动重定向到 `merged_to_node_id`
- 创建内容时：不允许选择已合并的节点

#### 3. 不处理的数据

**comment（节点评论）**
- 保留在源节点下
- **影响**：这些评论会随着源节点被屏蔽而无法访问
- **待确认问题3**：是否接受评论丢失？还是也需要迁移？

**content_stats（统计数据）**
- 不合并统计数据
- 源节点和目标节点的统计数据独立保持
- **说明**：因为评论不迁移，所以 comment_count 保持准确

**user_progress（学习进度）**
- 不更新 node_ids 文本字段
- **影响**：
  - 用户历史学习记录里还有源节点ID
  - 课程完成进度计算可能有偏差
- **说明**：学习进度是实时计算的，影响有限

**message（历史消息）**
- 不更新 content json 里的 nodeId
- **影响**：历史消息里的节点链接可能指向已合并的节点
- **处理**：前端显示时做兼容，检测到已合并节点时自动跳转

## 待确认问题汇总

### 问题1: user_card_srs 冲突处理
**场景**：
- 用户A在源节点下学习了卡片组X的卡片1，有SRS记录（id=100, user_id=1, card_id=10, node_id=123）
- 卡片组X被引用到目标节点，用户A在目标节点下也学习了卡片1，也有SRS记录（id=200, user_id=1, card_id=10, node_id=456）
- 批量更新后，两条记录的 node_id 都变成456

**问题**：是否会出现这种情况？如何处理？
- 选项A：这种情况不会出现（一个卡片组只能属于一个节点）
- 选项B：可能出现，需要去重（保留哪条？合并进度？）

### 问题2: 目录去重时的位置选择
**场景**：
```json
{
  "children": [
    {"nodeId": 123, "name": "Java基础"},
    {"nodeId": 456, "name": "其他内容"},
    {"nodeId": 456, "name": "Java进阶"}  // 目标节点
  ]
}
```
替换后：
```json
{
  "children": [
    {"nodeId": 456, "name": "Java基础"},  // 源节点变成目标节点
    {"nodeId": 456, "name": "其他内容"},
    {"nodeId": 456, "name": "Java进阶"}   // 原本就是目标节点
  ]
}
```

**问题**：去重时保留哪个位置的节点？
- 选项A：保留第一个出现的位置（删除后面的重复）
- 选项B：保留最后一个出现的位置（删除前面的重复）
- 选项C：都保留（不去重，允许目录中同一节点出现多次）

### 问题3: 节点评论是否迁移
**场景**：源节点下有100条评论（comment表中 object_type=node, object_id=源节点ID）

**问题**：是否迁移这些评论？
- 选项A：不迁移，评论随源节点一起被屏蔽（用户看不到了）
- 选项B：迁移，批量更新 comment.object_id（源节点ID → 目标节点ID）

如果选A，影响：
- 这些评论永久丢失（虽然数据还在，但无法访问）
- content_stats 的 comment_count 保持准确

如果选B，需要额外处理：
- 批量更新 comment 表
- content_stats 的 comment_count 需要重新计算或累加

## 技术实现概要

### 后端API设计
```
POST /api/v1/admin/nodes/merge
请求：
{
  "sourceNodeId": 123,
  "targetNodeId": 456
}

响应：
{
  "success": true,
  "mergedPostCount": 50,
  "mergedDeckCount": 3,
  "mergedSrsCount": 120,
  "updatedTocCount": 5
}
```

### 实现步骤
1. 验证源节点和目标节点都存在且有效
2. 开启数据库事务
3. 批量更新 post 表
4. 批量更新 memory_card_deck 表
5. 批量更新 user_card_srs 表
6. 更新所有引用该节点的 course_toc（解析JSON，替换ID，去重）
7. 更新所有引用该节点的 user_course_toc
8. 标记源节点状态为"已合并"，记录 merged_to_node_id
9. 提交事务
10. 记录操作日志（operation_log）
11. 异步：通知受影响的用户（可选）

### 前端拦截

**NodeDataService.java**
```java
public NodeDO validateAndGet(long nodeId) {
    NodeDO node = getById(nodeId);

    // 如果节点已合并，抛出异常或自动使用目标节点
    if (node.getState() == NodeState.MERGED.value()) {
        throw new NodeMergedException(node.getMergedToNodeId());
    }

    return node;
}
```

**前端路由守卫**
```typescript
// 访问已合并节点时自动重定向
if (nodeData.state === 'merged') {
  router.replace(`/read?nodeId=${nodeData.mergedToNodeId}`)
}
```

## 风险评估

### 低风险
- 文章迁移：简单的字段更新 ✅
- 卡片组迁移：简单的字段更新 ✅

### 中风险
- JSON目录更新：需要解析和去重，逻辑稍复杂
- SRS记录迁移：可能有冲突场景

### 需要注意
- **数据库事务**：必须保证原子性，要么全成功要么全失败
- **并发问题**：合并期间不能有新内容创建
- **回滚**：失败后如何恢复？需要记录操作前的状态吗？
