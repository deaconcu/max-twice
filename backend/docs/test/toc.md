# 课程目录管理接口测试用例

> 参考接口文档：`docs/api/toc.md`
> 测试类路径：`learn-web/src/test/java/com/prosper/learn/web/v1/controller/TocControllerTest.java`

---

## 测试覆盖概述

### 接口列表
1. **PUT** `/api/v1/users/current/courses/{courseId}/toc` - 更新用户课程目录
2. **GET** `/api/v1/users/current/courses/{courseId}/toc` - 获取用户课程目录

### 测试策略
- 使用 `@Transactional` 确保测试数据自动回滚
- 使用动态生成的 ID，避免硬编码
- 测试覆盖正常流程、参数验证、业务错误、边界条件
- 验证错误码与文档一致

---

## 测试数据准备

### 必需的测试辅助方法

需要创建以下辅助方法来准备测试数据：

1. **创建测试用户**
   - 功能：创建并返回一个测试用户
   - 参数：邮箱地址
   - 返回：UserDO 对象（包含自动生成的 ID）

2. **创建测试课程**
   - 功能：创建已发布的课程及其根节点
   - 参数：课程名称、创建者ID
   - 返回：CourseDO 对象
   - 注意：需要同时创建根节点并更新 course.rootNodeId

3. **创建用户课程目录**
   - 功能：为用户创建课程目录记录
   - 参数：用户ID、课程ID、目录哈希数组
   - 返回：UserCourseTocDO 对象

4. **创建课程目录版本**
   - 功能：创建课程目录内容记录
   - 参数：哈希值、目录JSON内容
   - 返回：CourseTocDO 对象

## 1. 更新用户课程目录

### 1.1 正常流程测试

#### 测试场景 1.1.1：使用现有哈希重新排列目录
**测试目标**：验证可以使用索引数组重新排列现有目录

**前置条件**：
- 创建测试用户
- 创建测试课程
- 为用户创建包含 3 个哈希的目录：`"hashA,hashB,hashC"`
- 创建对应的课程目录版本记录

**测试步骤**：
1. 用户登录
2. 发送 PUT 请求到 `/api/v1/users/current/courses/{courseId}/toc`
3. 请求体：`{"indexArray": "3,2,1"}`（反转顺序）

**预期结果**：
- HTTP 状态码：200
- 响应码：200
- 响应消息：`"目录更新成功"`
- 数据库验证：用户目录更新为 `"hashC,hashB,hashA"`

---

#### 测试场景 1.1.2：使用默认目录（索引为 0）
**测试目标**：验证索引值 0 会使用默认目录

**前置条件**：
- 创建测试用户和课程
- 为用户创建包含 2 个哈希的目录：`"hashA,hashB"`

**测试步骤**：
1. 用户登录
2. 发送 PUT 请求：`{"indexArray": "1,0,2"}`

**预期结果**：
- 响应码：200
- 数据库验证：
  - 用户目录更新为 `"hashA,defaultHash,hashB"`
  - 默认目录的 `refCount` 增加 1
  - 默认目录内容为：`{"rootNodeId": {}}`

---

#### 测试场景 1.1.3：全部使用默认目录
**测试目标**：验证可以将所有位置都重置为默认目录

**预期结果**：
- 响应码：200

- 默认目录的 `refCount` 增加 3

---

#### 测试场景 1.1.4：索引数组最大长度（9个元素）

## 注意事项

### 测试顺序
- 测试用例之间**不应有依赖关系**
- 每个测试独立准备数据
- 使用 `@Transactional` 确保隔离

### 数据准备
- 使用动态生成的 ID
- 避免硬编码 ID 值
- 测试数据应该最小化，只包含必需字段

### 断言验证
每个测试应验证：
1. HTTP 响应码
2. 业务错误码
3. 响应消息（如果有）
4. 数据库状态变化
5. 引用计数变化（如适用）

### Mock 使用
- **不推荐** Mock DataService 或 DomainService
- 使用真实的数据库操作（H2 或测试数据库）
- 只在必要时 Mock 外部服务

### 性能考虑
- 单个测试执行时间应 < 1 秒
- 使用 `@BeforeEach` 避免重复准备相同数据
- 批量创建测试数据时使用批量插入
