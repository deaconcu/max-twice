# 内容管理接口测试用例

> 参考测试编写规范：`docs/test/TEST_GUIDE.md`

---

## 测试准备

### 测试类基础配置

- 测试类：`ContentsControllerTest`
- 继承：`BaseControllerTest`
- 注解：`@Transactional`（测试后自动回滚）

### 依赖注入

需要注入以下服务：
- `MockMvc` - 用于发送 HTTP 请求
- `ObjectMapper` - 用于解析 JSON 响应
- `UserDomainService` - 创建测试用户
- `CourseDataService` - 创建测试课程
- `NodeDataService` - 创建测试节点
- `PostDataService` - 创建测试帖子
- `UserCourseTocDO` 相关服务 - 验证目录更新

### 测试辅助方法

需要以下辅助方法：
- `createUser(email)` - 创建测试用户
- `createPublishedCourse(name, creatorId)` - 创建已发布课程
- `createPublishedNode(name, courseId, creatorId)` - 创建已发布节点
- `createContentsPost(content, nodeId, creatorId)` - 创建内容型帖子（目录型）
- `createPublishedPost(content, nodeId, creatorId)` - 创建普通帖子
- `initUserCourseToc(userId, courseId)` - 初始化用户课程目录

## Command 测试（写操作）

### 1. 选择目录 (action=1)

#### 测试场景

##### 1.1 成功选择目录
- **准备**：
  - 创建用户A、课程、节点
  - 创建内容型帖子（包含有效的目录 JSON）
  - 初始化用户的课程目录
  - 用户A登录
- **请求**：POST /api/v1/contents，action=1，传入 path、courseId、postingId
- **验证**：
  - 返回 200 状态码
  - 用户课程目录已更新
  - 目录中包含选中标记（`+` 字段指向该帖子ID）

##### 1.2 字段验证 - path 为空
- **准备**：用户登录
- **请求**：path=""
- **验证**：返回 1002 (INVALID_PARAMETER)，提示"路径不能为空"

##### 1.3 字段验证 - path 缺失
- **准备**：用户登录
- **请求**：不传 path 字段
- **验证**：返回 1002 (INVALID_PARAMETER)

##### 1.4 字段验证 - courseId 为 null
- **准备**：用户登录
- **请求**：不传 courseId
- **验证**：返回 1002 (INVALID_PARAMETER)，提示"课程ID不能为空"

##### 1.5 字段验证 - courseId = 0
- **准备**：用户登录
- **请求**：courseId=0
- **验证**：返回 1002 (INVALID_PARAMETER)，提示"课程ID必须为正数"

##### 1.6 字段验证 - courseId = -1
- **准备**：用户登录
- **请求**：courseId=-1
- **验证**：返回 1002 (INVALID_PARAMETER)

##### 1.7 字段验证 - postingId 为 null
- **准备**：用户登录
- **请求**：不传 postingId
- **验证**：返回 1002 (INVALID_PARAMETER)，提示"帖子ID不能为空"

##### 1.8 字段验证 - postingId = 0
- **准备**：用户登录
- **请求**：postingId=0
- **验证**：返回 1002 (INVALID_PARAMETER)

##### 1.9 字段验证 - action 为 null
- **准备**：用户登录
- **请求**：不传 action
- **验证**：返回 1002 (INVALID_PARAMETER)，提示"操作类型不能为空"

##### 1.10 字段验证 - action = 0
- **准备**：用户登录
- **请求**：action=0
- **验证**：返回 1002 (INVALID_PARAMETER)，提示"操作类型不正确"

##### 1.11 字段验证 - action = 5
- **准备**：用户登录
- **请求**：action=5（超出范围）
- **验证**：返回 1002 (INVALID_PARAMETER)

##### 1.12 业务验证 - 课程不存在
- **准备**：用户登录
- **请求**：courseId=99999（不存在的课程）
- **验证**：返回 1007 (NOT_FOUND)，提示"课程不存在"

##### 1.13 业务验证 - 帖子不存在
- **准备**：用户登录，创建课程
- **请求**：postingId=99999（不存在的帖子）
- **验证**：返回 1007 (NOT_FOUND)，提示"帖子不存在"

##### 1.14 业务验证 - 用户目录不存在
- **准备**：用户登录，创建课程，但不初始化用户目录
- **请求**：正常的选择目录请求
- **验证**：返回 1007 (NOT_FOUND)，提示"用户课程目录不存在"

##### 1.15 业务验证 - 帖子类型错误
- **准备**：用户登录，创建普通帖子（非内容型）
- **请求**：尝试选择该普通帖子作为目录
- **验证**：返回业务错误（帖子类型不匹配）

##### 1.16 权限验证 - 未登录
- **请求**：不传 token
- **验证**：返回 1101 (USER_NOT_LOGIN)

## 测试数据准备

### 基础数据

每个测试需要准备：
1. **用户**: 至少1个测试用户
2. **课程**: 已发布的课程（包含根节点）
3. **节点**: 课程下的测试节点
4. **帖子**:
   - 内容型帖子（用于选择目录测试）
   - 普通帖子（用于固定/取消固定测试）
5. **用户目录**: 初始化用户的课程目录

### 内容型帖子结构

用于测试选择目录的帖子内容示例：
```
节点123,节点456,节点789
```

格式说明：
- 逗号分隔的节点ID列表
- 会被解析为目录结构

### 路径格式

测试中使用的路径示例：
- `"1-123"` - tocIndex=1, nodeId=123
- `"2-456"` - tocIndex=2, nodeId=456

## 重要测试点

### 1. 参数验证完整性

需要验证所有参数的边界情况：
- **path**: 空字符串、null、格式错误
- **courseId**: null、0、负数、不存在
- **postingId**: null、0、负数、不存在
- **action**: null、0、5（超出范围）

### 2. 操作类型区分

确保4种操作正确执行：
- action=1 调用 `tocService.choose()`
- action=2 调用 `tocService.unchoose()`
- action=3 调用 `tocService.pin(..., true)`
- action=4 调用 `tocService.pin(..., false)`

### 3. 业务规则验证

#### 选择目录（action=1）
- 只能选择内容型帖子（PostType.CONTENTS）
- 帖子内容必须是有效的目录格式
- 用户目录必须已初始化

#### 固定帖子（action=3）
- 可以固定任意类型的帖子
- 固定数量有上限（配置决定）
- 重复固定不报错（幂等）

#### 取消操作（action=2/4）
- 取消不存在的选择/固定不报错
- 幂等性保证

### 4. 目录更新验证

每次操作后需要验证：
- 用户课程目录的哈希值是否变化
- 目录内容是否包含正确的标记
- 引用计数是否正确更新

### 5. 权限控制

- 所有操作都需要登录
- 只能操作自己的目录
- 不能操作他人的目录（即使是同一课程）

## 边界场景测试

### 1. 路径格式验证
- 有效路径：`"1-123"`, `"9-999999"`
- 无效路径：`"abc"`, `"0-123"`, `"10-123"`, `"-123"`, `"1-"`

### 2. tocIndex 边界
- 最小值：1
- 最大值：9
- 超出范围：0, 10, 负数

### 3. 固定帖子上限
- 达到上限前：可以继续固定
- 达到上限后：返回错误
- 取消固定后：可以再次固定

### 4. 并发操作
- 同一用户对同一课程的并发操作
- 不同用户对同一课程的并发操作（应该互不影响）

## 数据库验证点

### 选择目录后
检查 `user_course_toc` 表：
- `toc` 字段已更新（哈希值变化）
- `updated_at` 时间已更新

检查 `course_toc` 表：
- 新增一条目录版本记录（如果是新结构）
- 或者引用计数+1（如果是已有结构）

### 固定帖子后
检查目录内容：
- `^` 字段包含固定的帖子ID
- 多个帖子用逗号分隔

## 注意事项

1. **@Transactional 自动回滚**:
   - 每个测试执行完后，数据库操作自动回滚
   - 无需手动清理测试数据
   - 但 Redis 中的数据（如限流计数）不会回滚

2. **动态ID**:
   - 所有测试数据使用数据库自动生成的ID
   - 避免硬编码ID，防止测试冲突

3. **认证状态管理**:
   - 使用 `StpUtil.login(userId)` 模拟登录
   - 测试结束在 finally 块中调用 `StpUtil.logout()`
   - 确保登录状态不会影响其他测试

4. **目录初始化**:
   - 测试前需要初始化用户课程目录
   - 可以通过调用页面聚合接口自动初始化
   - 或者直接插入 `user_course_toc` 记录

5. **帖子内容格式**:
   - 内容型帖子的 content 字段需要是有效的节点ID列表
   - 格式：`"123,456,789"`（逗号分隔）
   - 用于测试选择目录功能

6. **固定数量限制**:
   - 配置项：`app.contents.max-pinned-items`
   - 测试时需要知道该配置值
   - 测试固定数量上限场景

7. **路径解析**:
   - 路径格式：`tocIndex-nodeId`
   - tocIndex 范围：1-9
   - 错误的格式会导致解析失败

8. **目录版本管理**:
   - 每次修改目录都会生成新的哈希版本
   - 相同结构的目录共享哈希值
   - 引用计数自动管理

---

## 测试执行顺序建议

1. 先测试参数验证（快速失败）
2. 再测试权限验证（未登录场景）
3. 然后测试业务验证（数据不存在）
4. 最后测试正常场景和业务规则

这样可以快速发现基础问题，避免浪费时间在复杂场景上。
