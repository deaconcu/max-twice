# 记忆卡片接口测试文档

> 测试类: `MemoryCardControllerTest`
> 测试目标: 验证记忆卡片 CRUD 接口的正确性、权限控制和 SRS 学习状态管理

---

## 测试环境准备

### 依赖组件
- MockMvc（用于接口测试）
- MemoryCardService（卡片服务）
- MemoryDeckService（卡片组服务）
- UserService（用户服务）

### 测试数据准备
- 创建测试用户（卡片组创建者、普通用户）
- 创建测试卡片组（已发布状态）
- 创建测试卡片（包含不同 SRS 状态的卡片）
- 创建测试节点和课程

### 数据清理
- 使用 @Transactional 自动回滚数据库数据
- 每个测试前清理关联数据

## 一、创建卡片接口测试

### 1.1 成功创建卡片
- 登录用户，提供有效的 deckId、front、back
- 验证返回 200 状态码
- 验证返回 code 为 200
- 验证返回 message 为 "创建成功"
- 验证数据库中新增了卡片记录
- 验证自动创建了初始 SRS 状态（type=0 NEW）

### 1.2 缺少必填参数
- 不提供 deckId
- 验证返回 1002 参数异常
- 不提供 front
- 验证返回 1002 参数异常
- 不提供 back
- 验证返回 1002 参数异常

### 1.3 字段长度验证
- front 超过最大长度
- 验证返回 1002 参数异常
- back 超过最大长度
- 验证返回 1002 参数异常
- front 为空字符串
- 验证返回 1002 参数异常

### 1.4 卡片组不存在
- 提供不存在的 deckId
- 验证返回 2201 卡片组不存在

### 1.5 未登录
- 不提供 token
- 验证返回 1101 用户未登录

### 1.6 限流测试
- 短时间内发送超过 50 次请求
- 验证返回 2301 访问过于频繁

## 四、获取卡片内容差异接口测试

### 4.1 有差异 - 用户版本与卡片组版本不同
- 用户修改了卡片内容
- 验证返回 hasDiff = true
- 验证返回 deckVersion（卡片组原始版本）
- 验证返回 userVersion（用户修改版本）

- 用户未修改卡片，或已同步到最新版本
- 验证返回 hasDiff = false
- 验证返回提示消息 "卡片内容与卡片组一致"

### 4.3 卡片不存在

- 验证返回 2202 卡片不存在

## 测试注意事项

1. **权限测试**
   - 每个接口都需要测试已登录和未登录场景
   - 更新和删除接口需要测试权限控制（只有创建者可操作）

2. **数据一致性**
   - 创建卡片后验证 SRS 状态也被创建
   - 删除卡片后验证 SRS 状态也被删除
   - 更新卡片后验证 hasCardUpdate 标记正确

3. **限流测试**
   - 所有接口限流为 50 次/分钟/用户
   - 使用独立的测试方法验证限流机制

4. **SRS 状态**
   - 验证不同状态的 interval 单位正确（分钟 vs 天）
   - 验证状态流转规则正确

5. **软删除**
   - 删除操作是软删除，验证 deleted_at 字段
   - 软删除后的卡片不应出现在查询结果中

## Mock 策略

### 需要 Mock 的组件
- 无（使用真实的 Service 和数据库）

### 不需要 Mock 的组件
- MemoryCardService（使用真实实例）
- MemoryDeckService（使用真实实例）
- 数据库（使用测试数据库，@Transactional 自动回滚）

### Mock 建议
- 建议使用真实数据库进行集成测试
- 使用 @Transactional 确保测试数据自动清理
- 如需测试数据库异常场景，可以使用 Mock
