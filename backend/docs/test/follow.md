# 关注管理接口测试文档 (FollowsController)

## 测试文件

- **测试类**: `FollowsControllerTest.java`
- **位置**: `learn-web/src/test/java/com/prosper/learn/web/v1/controller/`
- **API文档**: `docs/api/follow.md`
- **Controller**: `FollowsController.java`

## 测试框架配置

- 使用 `@SpringBootTest` 和 `@AutoConfigureMockMvc`
- 使用 `@Transactional` 实现测试后自动回滚
- 继承 `BaseControllerTest`

### 依赖注入

需要注入以下服务：
- `MockMvc` - 用于发送 HTTP 请求
- `WebApplicationContext` - Web 应用上下文
- `FollowDataService` - 关注数据服务
- `UserDataService` - 用户数据服务
- `UserDomainService` - 用户领域服务（用于创建测试用户）
- `ObjectMapper` - JSON 解析

---

## 测试数据准备

### 必需的测试辅助方法

需要创建以下辅助方法来准备测试数据：

1. **创建测试用户**
   - 功能：创建并返回一个测试用户
   - 参数：邮箱地址、密码
   - 返回：UserDO 对象（包含自动生成的 ID）

2. **创建关注关系**
   - 功能：创建关注者与被关注者的关系记录
   - 参数：关注者ID、被关注者ID
   - 返回：FollowDO 对象

3. **验证关注关系存在**
   - 功能：验证数据库中是否存在指定的关注记录
   - 参数：关注者ID、被关注者ID
   - 返回：boolean

## 1. 关注用户接口 (POST /api/v1/follows)

### 1.1 正常流程测试

#### 测试场景 1.1.1：首次关注用户成功
**测试目标**：验证用户可以成功关注另一个用户

**前置条件**：
- 创建两个测试用户：user1（关注者）、user2（被关注者）
- user1 和 user2 之间没有关注关系
- user1 已登录

**测试步骤**：
1. user1 登录系统
2. 发送 POST 请求到 `/api/v1/follows`
3. 请求体：`{"followeeId": user2.id}`

**预期结果**：
- HTTP 状态码：200
- 响应码：200
- 响应消息：`"操作成功"`
- 无 data 字段
- 数据库验证：
  - 存在关注记录（follower_id=user1.id, followee_id=user2.id）
  - 记录了关注时间（created_at）

**事件验证**：
- 发布了 `UserFollowedEvent` 事件
- 事件数据：followerId=user1.id, followeeId=user2.id

---

#### 测试场景 1.1.2：重复关注（幂等性测试）
**测试目标**：验证重复关注同一用户不会报错

**预期结果**：
- HTTP 状态码：200

- 数据库验证：

**事件验证**：
- 未发布新的 `UserFollowedEvent` 事件

**预期结果**：
- HTTP 状态码：200

---

#### 测试场景 3.1.2：获取第一页（有数据）
**测试目标**：验证正确返回关注列表

**前置条件**：
- 创建 user1、user2、user3、user4
- user1 关注了 user2、user3、user4（按此顺序）
- user2 有头像
- user3 有个人简介
- user4 头像为 null

**测试步骤**：
1. 发送 GET 请求到 `/api/v1/users/{user1.id}/followees`

**预期结果**：
- HTTP 状态码：200
- 响应码：200
- data 数组包含 3 个元素
- 排序：按关注时间降序（最新关注的在前）

  - name：被关注用户的姓名

## 10. 注意事项

1. **幂等性测试重要性**
   - 关注和取消关注都需要测试重复操作的幂等性
   - 验证重复操作不会导致数据不一致或错误

2. **事件验证**
   - 测试时需要验证领域事件是否正确发布
   - 可以使用 `@EventListener` 或 mock 来捕获事件

3. **时间相关测试**
   - 涉及时间的测试需要考虑时区问题
   - 时间格式必须与文档一致：`yyyy-MM-dd HH:mm:ss`

4. **分页测试**
   - 验证时间游标分页的正确性
   - 确保没有数据重复或遗漏

5. **并发测试**
   - 使用 `CountDownLatch` 或类似工具模拟并发
   - 验证数据库事务隔离级别是否正确

6. **性能测试**
   - 批量查询优化很重要
   - 应该验证 N+1 查询问题已解决
