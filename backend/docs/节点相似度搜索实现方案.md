# 节点相似度搜索实现方案

## 背景

为了防止用户创建重复节点，需要在创建节点时实时检测相似的已存在节点，引导用户优先使用现有节点。

## 技术方案

### 整体架构

使用三个组件配合：
1. **MySQL**：存储节点基本信息（id, name, description）
2. **OpenAI API**：将文本转换为向量（embedding）
3. **Milvus**：存储向量并提供快速相似度搜索

### 数据流向

**创建节点**：
- 节点信息存入MySQL
- 标题+描述发送给OpenAI生成向量
- 向量和节点ID存入Milvus
- 三者通过节点ID关联

**搜索相似节点**：
- 用户输入文本发送给OpenAI生成查询向量
- 用查询向量在Milvus中搜索最相似的节点ID
- 用节点ID列表从MySQL查询完整信息
- 返回给前端展示

---

## 技术选型

### AI Embedding 服务：OpenAI

**模型**：text-embedding-3-small
**向量维度**：1536维
**为什么选它**：
- 中文支持好
- 价格便宜（$0.02/1M tokens）
- API稳定可靠

**成本估算**：
- 初始化8000个节点：约$0.16
- 每天新增100个节点：约$0.002
- 每天1000次查询：约$0.02
- **月成本**：不到$1

### 向量数据库：Milvus

**为什么选Milvus而不是ES**：
- 现在只需要向量搜索，不需要全文搜索
- Milvus更轻量，部署简单
- 专业向量数据库，性能更好
- 将来迁移到ES也容易

**部署方式**：
- Docker Compose（最简单）
- 或使用云服务（阿里云、Zilliz Cloud）

**性能**：10万节点搜索耗时 50-200ms

## 数据存储设计

### MySQL 节点表
不需要改动，保持原样。不存储embedding（embedding存在Milvus）。

### Milvus Collection
创建一个名为 `node_embeddings` 的集合：
- 字段1：id（节点ID，主键）
- 字段2：embedding（1536维浮点向量）
- 索引：HNSW算法，余弦相似度

---

## 功能实现

### 1. 服务层设计

**EmbeddingService**：
- 职责：调用OpenAI API生成向量
- 方法：generateEmbedding(text) 返回向量数组

**MilvusService**：
- 职责：向量的增删改查
- 方法：
  - insert(nodeId, embedding) - 插入向量
  - search(queryEmbedding, topK, threshold) - 搜索相似向量
  - update(nodeId, embedding) - 更新向量
  - delete(nodeId) - 删除向量

**NodeApplicationService**：
- 整合业务逻辑
- 创建节点时自动生成embedding并同步到Milvus
- 提供相似节点搜索接口

### 2. 增量更新机制

**节点创建时**：
- 保存到MySQL后，异步生成embedding并存入Milvus
- 失败不影响节点创建，通过补偿机制处理

**节点更新时**：
- 检查标题或描述是否变化
- 如果变化，重新生成embedding并更新Milvus
- 如果没变化，不重新生成（节省成本）

**节点删除时**：
- MySQL软删除
- Milvus也删除对应向量

### 3. API接口设计

**搜索相似节点**：
- 路径：GET /api/v1/nodes/search-similar
- 参数：query（查询文本）、limit（返回数量，默认10）、threshold（相似度阈值，默认0.85）
- 返回：相似节点列表（包含相似度分数）

**初始化embedding**（管理员接口）：
- 路径：POST /api/v1/admin/nodes/init-embeddings
- 功能：批量为现有节点生成embedding
- 返回：处理进度和结果统计

### 4. 前端集成

**搜索模式**：
- 用户点击搜索按钮
- 调用 searchSimilar API
- 展示结果列表，每个结果显示相似度百分比

**创建模式**：
- 用户输入节点名称时（防抖500ms）
- 调用 searchSimilar API（阈值0.9，只返回3个）
- 如果找到相似节点，显示警告提示

---

## 相似度阈值说明

**推荐阈值设置**：
- **0.95以上**：几乎相同，强烈建议使用现有节点
- **0.85-0.95**：很相似，建议使用现有节点
- **0.75-0.85**：有点相似，可以参考
- **0.75以下**：不相似，可以忽略

**实际使用**：
- 创建模式的相似检测：阈值0.9（只提示非常相似的）
- 搜索模式的结果展示：阈值0.85（展示更多可能性）

## 容错和补偿机制

### 失败场景处理

**OpenAI API 调用失败**：
- 记录失败日志
- 定时任务扫描未生成embedding的节点
- 重试生成

**Milvus 连接失败**：
- 降级处理：不影响节点创建
- 记录待同步队列
- 恢复后批量补偿

### 监控指标

- embedding生成成功率
- Milvus同步成功率
- 搜索响应时间
- API调用成本

## 部署和初始化

### 部署步骤

1. 部署Milvus（Docker或云服务）
2. 配置OpenAI API Key
3. 应用启动时自动创建Milvus Collection
4. 调用管理接口初始化现有8000个节点的embedding

### 初始化注意事项

- 批量处理，每次100个节点
- 控制并发，避免超过OpenAI速率限制
- 记录进度，支持断点续传
- 预计耗时：约1-2小时（8000个节点）
