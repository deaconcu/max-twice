# 学习进度系统设计文档

## 一、系统概述

这是一个基于**双表设计**的学习进度管理系统，采用 DDD（领域驱动设计）架构和事件驱动模式。系统的核心特点是：

- **双表分离**：学习状态和具体完成记录分开管理
- **覆盖算法**：父节点完成自动覆盖子节点，提供灵活的学习路径
- **事件驱动**：ToC 变更自动维护反向查找缓存
- **实时计算**：避免数据不一致，基于当前状态实时计算进度
- **批量优化**：多级别的批量查询优化性能
- **统一抽象**：课程和路线图使用同一套学习记录机制

## 二、核心表结构

### 2.1 `user_learning` 表（学习记录统一表）

**职责**：记录用户开始学习某个课程/节点/路线图的状态和进度

```sql
CREATE TABLE `user_learning` (
    `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    `user_id` BIGINT UNSIGNED NOT NULL COMMENT '用户ID',
    `object_type` TINYINT UNSIGNED NOT NULL COMMENT '对象类型：2=node，4=roadmap',
    `object_id` BIGINT UNSIGNED NOT NULL COMMENT '对象ID：对于课程是rootNodeId，对于路线图是roadmapId',
    `parent_id` BIGINT UNSIGNED DEFAULT NULL COMMENT '父对象ID：路线图使用（professionId），节点不使用',
    `is_root_node` TINYINT(1) NOT NULL DEFAULT 0 COMMENT '是否为课程根节点：0=普通节点/路线图，1=课程根节点',
    `progress_percent` INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '进度（0-10000，万分位精度）',
    `nodes` TEXT DEFAULT NULL COMMENT '节点ID列表（JSON数组格式）',
    `started_at` DATETIME DEFAULT NULL COMMENT '开始学习时间',
    `completed_at` DATETIME DEFAULT NULL COMMENT '完成时间',
    `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_user_object` (`user_id`, `object_type`, `object_id`),
    KEY `idx_user_type` (`user_id`, `object_type`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户学习记录表';
```

**关键字段说明**：

- **object_type + object_id + is_root_node**：
  - 课程学习：`object_type=2, object_id=rootNodeId, is_root_node=1`
  - 普通节点学习：`object_type=2, object_id=nodeId, is_root_node=0`
  - 路线图学习：`object_type=4, object_id=roadmapId, is_root_node=0`

- **nodes**：JSON 数组格式，存储用户当前选择的第一个目录的所有节点 ID
  - 格式示例：`"[101,102,103,104,105]"`
  - 用途：快速反向查找（节点完成时，哪些课程受影响）
  - 维护时机：用户选择/切换目录时自动更新

- **progress_percent**：万分位精度（0-10000）
  - 通过递归算法实时计算
  - 基于用户当前目录和节点完成状态

- **state**：虽然数据库中不存储，但通过 `getState()` 方法动态计算
  - `progress_percent >= 10000` → 已完成 (2)
  - `0 < progress_percent < 10000` → 进行中 (1)

### 2.2 `user_node_completion` 表（节点完成记录表）

**职责**：记录用户标记"学会"的节点（可以是叶子节点或目录节点）

```sql
CREATE TABLE `user_node_completion` (
    `user_id` BIGINT UNSIGNED NOT NULL,
    `node_id` BIGINT UNSIGNED NOT NULL,
    `completed_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (`user_id`, `node_id`),
    KEY `idx_user` (`user_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户节点完成记录表';
```

**特点**：
- 每个完成的节点一条记录
- 可以是叶子节点或目录节点（任何节点）
- 跨课程、跨目录持久化
- 用于计算课程进度的依据

### 2.3 两个表的关系

```
user_node_completion (节点的绝对完成状态)
        ↓ 用于计算
user_learning (课程/目录的学习进度)
```

**关键理解**：
- **如果节点 A 在 user_node_completion 中被标记"学会"**
  - → 以节点 A 为根的任何课程/目录，进度都是 100%（覆盖效应）

- **以节点 A 为根的课程进度 100%**
  - ≠ 节点 A 本身被标记"学会"
  - 可能是所有子节点都学会了（递归计算得出）

### 3.2 用户选择/切换目录

```
用户操作：选择目录中的某个节点

后端流程：
1. TocDomainService.choose() / unchoose() / updateUserNodeToc()
   ↓
2. 检测到第一个目录发生变化
   ↓
3. 发布 TocChosenEvent(userId, nodeId, newFirstTocHash)
   ↓
4. TocEventListener.onTocChosen() 监听事件
   ↓

   WHERE user_id=? AND object_type=2 AND object_id=nodeId
   ↓
6. 检查是否需要更新：
   - 如果未开始学习 → 跳过
   - 如果已完成（progress >= 10000）→ 跳过
   ↓
7. 获取新目录内容（通过 tocHash）
   ↓
8. 解析 ToC JSON，提取所有节点 ID
   ↓
9. 更新 user_learning.nodes 字段：
   nodes = "[101,102,103,104,105]"
   ↓
10. 重新计算进度并更新 progress_percent
```

### 3.3 标记节点"学会"（核心流程）

```
用户操作：点击节点旁的"学会"按钮

前端 API 调用：
POST /api/v1/progress/nodes/{nodeId}/complete
Body: { courseId }

后端流程：
1. ProgressController.markNodeCompleted(nodeId, courseId)
   ↓
2. LearningProgressService.markNodeCompletedWithResponse()
   ↓
3. 验证节点和课程存在
   ↓
4. LearningProgressDomainService.markNodeCompleted(userId, nodeId)
   ├─ 检查是否已完成（防止重复标记）
   ├─ 如果已完成 → 抛出 NODE_ALREADY_COMPLETED 异常
   └─ 插入 user_node_completion 表：
      INSERT INTO user_node_completion (user_id, node_id) VALUES (?, ?)
   ↓
5. updateAffectedCoursesProgress(userId, nodeId)
   ├─ 查询所有包含该节点的课程：
   │  SELECT * FROM user_learning
   │  WHERE user_id=?
   │  AND object_type=2
   │  AND nodes LIKE '%nodeId%'
   │
   └─ 遍历每门课程：
      ├─ updateSingleCourseProgress(userId, learning)
      │  ├─ 获取用户第一个目录的 ToC
      │  ├─ calculateNodeProgress(userId, rootNodeId)
      │  │  └─ 递归计算进度（见下文算法）
      │  ├─ oldProgress = learning.progress_percent
      │  ├─ 更新 user_learning.progress_percent = newProgress
      │  └─ 如果 oldProgress < 10000 && newProgress >= 10000：
      │     └─ 自动设置 completed_at
      │     └─ 发布 LearningCompletedEvent（可选）
      │
      └─ 重复处理下一门课程
   ↓
6. 返回 NodeProgressResponseDTO { nodeId, completed: true }
```

### 3.4 进度计算算法（递归+覆盖）

```
calculateNodeProgress(userId, nodeId):
  ↓
1. 获取用户第一个目录的 ToC 内容
   tocContent = tocService.getToc(userId, nodeId, 1)
   ↓
2. 解析 ToC JSON
   tocNode = objectMapper.readTree(tocContent)
   ↓
3. 收集目录中的所有节点 ID
   nodeIdsInToc = collectNodeIds(tocNode)
   // 包括所有层级的节点（目录节点+叶子节点）
   ↓
4. 批量查询这些节点的完成状态
   completedNodes = domainService.getCompletedNodesInList(userId, nodeIdsInToc)
   // 返回所有在 user_node_completion 表中的节点ID集合
   ↓
5. 递归计算层级进度
   progress = calculateHierarchicalProgress(tocNode, completedNodes)
   ↓
6. 返回进度（0-10000）

---

递归算法详解：

calculateHierarchicalProgress(node, completedNodes):

  // 1. 收集所有子节点
  for each field in node:
      if field 是数字:
          childNodeIds.add(field)
          childNodes.add(node.get(field))

  if childNodeIds 为空:
      return 0.0

  // 2. 遍历每个子节点计算完成度
  totalProgress = 0.0

  for each (nodeId, childNode) in (childNodeIds, childNodes):

      // 【关键：覆盖逻辑】
      // 先检查节点本身是否已完成
      if nodeId in completedNodes:
          // 节点已完成 → 完成度 100%
          // 不再递归计算子节点（所有子节点被覆盖）
          totalProgress += 1.0
          continue

      // 节点未完成，检查是否有子节点
      if childNode 有子节点:
          // 递归计算子节点的平均完成度
          childProgress = calculateHierarchicalProgress(childNode, completedNodes)
          totalProgress += childProgress
      else:
          // 叶子节点且未完成
          totalProgress += 0.0

  // 3. 返回平均进度
  return totalProgress / childNodeIds.size()
```

**算法示例**：

```
目录结构：
根节点 (ID=100)
├── 第1章 (ID=10) ✓ 已完成
│   ├── 1.1 (ID=101) ✓ 已完成
│   └── 1.2 (ID=102) ○ 未完成
└── 第2章 (ID=20) - 未完成
    ├── 2.1 (ID=201) ✓ 已完成
    └── 2.2 (ID=202) ○ 未完成

completedNodes = {10, 101, 201}

计算过程：

calculateHierarchicalProgress(根节点):
  遍历子节点 [第1章, 第2章]

  子节点: 第1章 (ID=10)
    → 10 in completedNodes? YES
    → totalProgress += 1.0  // 100%（覆盖，不看子节点101,102）

  子节点: 第2章 (ID=20)
    → 20 in completedNodes? NO
    → 有子节点，递归计算
    → calculateHierarchicalProgress(第2章):
        遍历子节点 [2.1, 2.2]

        子节点: 2.1 (ID=201)
          → 201 in completedNodes? YES
          → totalProgress += 1.0

        子节点: 2.2 (ID=202)
          → 202 in completedNodes? NO
          → 是叶子节点且未完成
          → totalProgress += 0.0

        return (1.0 + 0.0) / 2 = 0.5  // 50%

    → totalProgress += 0.5

  return (1.0 + 0.5) / 2 = 0.75  // 75%

最终进度：75% × 10000 = 7500 (万分位)
```

## 四、进度计算的"覆盖"机制详解

### 4.1 覆盖机制的核心思想

**概念**：当用户标记一个父节点为"学会"时，其所有子节点的完成状态在进度计算时被"覆盖"。

**数据层面**：
- 子节点的完成记录依然保留在 `user_node_completion` 表中
- 只是在计算时被忽略

**用户体验**：
- 绿色 ✓ = 节点已完成，且计入进度
- 灰色 ✓ = 节点已完成，但被父节点覆盖，不计入进度
- 圆圈 ○ = 叶子节点未完成
- 横杠 - = 目录节点未完成

### 4.2 覆盖机制的实际应用

**场景1：标记父节点完成**

```
操作前：
第1章 - (未完成)
├── 1.1 ✓ (已完成)
└── 1.2 ○ (未完成)
进度：50%

user_node_completion: [101]

用户点击"第1章"的"学会"按钮
↓
操作后：
第1章 ✓ (已完成，绿色)
├── 1.1 ✓(灰) (被覆盖)
└── 1.2 ○(灰) (被覆盖)
进度：100%

user_node_completion: [101, 10]  // 追加第1章，不删除子节点

进度计算：
检测到第1章(ID=10)已完成 → 返回100%，不再检查1.1和1.2
```

**场景2：取消父节点完成**

```
用户点击"第1章"取消"学会"
↓
第1章 - (未完成)
├── 1.1 ✓ (恢复正常显示)
└── 1.2 ○ (恢复正常显示)
进度：50%

user_node_completion: [101]  // 删除第1章，保留1.1

进度计算：
第1章未完成 → 递归计算子节点 → (1.0 + 0.0) / 2 = 50%
```

**场景3：多层嵌套覆盖**

```
根节点 ✓ (已完成)
└── 第1章 ✓ (已完成，但被根节点覆盖，显示灰色)
    └── 1.1 ✓ (已完成，但被根节点覆盖，显示灰色)

user_node_completion: [100, 10, 101]

进度计算：
检测到根节点(ID=100)已完成 → 直接返回100%
不再检查第1章和1.1

取消根节点完成：
根节点 - (未完成)
└── 第1章 ✓ (恢复绿色)
    └── 1.1 ✓(灰) (被第1章覆盖)

进度计算：
根节点未完成 → 递归
检测到第1章已完成 → 返回100%
```

### 4.3 权重分配机制

**原则**：每层兄弟节点平分父节点权重

```
根节点 (权重 = 1.0)
├── 子节点1 (权重 = 1/4 = 0.25)
│   ├── 叶子A (权重 = 0.25/2 = 0.125)
│   └── 叶子B (权重 = 0.25/2 = 0.125)
│
├── 子节点2 (权重 = 1/4 = 0.25)
│   ├── 叶子C (权重 = 0.25/3 = 0.0833)
│   ├── 叶子D (权重 = 0.25/3 = 0.0833)
│   └── 叶子E (权重 = 0.25/3 = 0.0833)
│
├── 子节点3 (权重 = 1/4 = 0.25) [未选目录]
└── 子节点4 (权重 = 1/4 = 0.25) [未选目录]

如果用户学会了：叶子A, 叶子B, 叶子C, 叶子D
计算：
子节点1 = (1.0 + 1.0) / 2 = 1.0 = 100%
子节点2 = (1.0 + 1.0 + 0.0) / 3 = 0.6667 = 66.67%
子节点3 = 0% (未选目录，视为叶子节点未完成)
子节点4 = 0%

根节点 = (1.0 + 0.6667 + 0.0 + 0.0) / 4 = 0.4167 = 41.67%

对应你的公式：1/4 + 1/4 × 2/3 = 0.25 + 0.1667 = 0.4167 ✓
```

### 4.4 覆盖机制的数据一致性

**关键**：子节点的完成记录永久保留

```
时间线：
T1: 标记叶子101, 102完成
    user_node_completion: [101, 102]

T2: 标记父节点10完成
    user_node_completion: [101, 102, 10]  // 追加，不删除
    进度计算：检测到10完成 → 100%

T3: 在父节点完成期间，标记叶子103完成
    user_node_completion: [101, 102, 10, 103]
    进度计算：还是100%（被父节点10覆盖）

T4: 取消父节点10完成
    user_node_completion: [101, 102, 103]  // 删除10，保留所有子节点
    进度计算：恢复基于子节点 → (1+1+1) / 总数
```

## 六、事件驱动架构

### 6.1 事件类型

#### LearningStartedEvent
**触发时机**：用户开始学习课程/路线图
```java
@Data
@AllArgsConstructor
public class LearningStartedEvent {
    private Long userId;
    private Long contentId;  // courseId 或 roadmapId
    private ContentType contentType;
}
```

**发布位置**：
- `UserLearningService.startLearningCourse()`
- `UserLearningService.startLearning()`

**当前监听器**：无

#### LearningCompletedEvent
**触发时机**：课程/路线图进度达到 100%

**当前状态**：定义了事件，但没有监听器，暂不发布

**未来用途**：

### 6.2 TocEventListener 详解

```java
@EventListener
public void onTocChosen(TocChosenEvent event) {
    Long userId = event.getUserId();
    Long nodeId = event.getNodeId();
    String newFirstTocHash = event.getNewFirstTocHash();

    // 1. 查询用户是否在学习这个节点
    UserLearningDO learning = userLearningDataService.getByUserAndObject(
        userId, Enums.ContentType.node.byteValue(), nodeId
    );

    // 2. 如果未开始学习或已完成，跳过更新
    if (learning == null) {
        return;
    }

    if (learning.getState() == UserProgressState.COMPLETED.value()) {
        return;  // 已完成不需要更新
    }

    // 3. 获取新目录内容
    NodeTocDO nodeTocDO = nodeTocDataService.get(newFirstTocHash);
    String tocContent = nodeTocDO.getToc();

    // 4. 解析 ToC，提取所有节点ID
    Set<Long> nodeIds = collectNodeIds(tocContent);
    String nodesJson = "[" + String.join(",", nodeIds) + "]";

    // 5. 更新 user_learning.nodes 字段
    userLearningDataService.updateNodes(userId, node.byteValue(), nodeId, nodesJson);

    // 6. 重新计算进度
    Integer newProgress = learningProgressService.calculateNodeProgress(userId, nodeId);
    userLearningDomainService.updateProgress(userId, ContentType.node, nodeId, newProgress);
}
```

**作用**：
- 自动维护 `user_learning.nodes` 字段
- 确保反向查找的准确性
- 目录变化时自动重新计算进度

## 七、性能优化策略

### 7.1 批量查询优化

**批量计算进度**：
```java
batchCalculateNodeProgress(userId, nodeIds)
  ↓
1. 批量获取所有节点的 ToC
   Map<Long, String> tocMap = tocService.batchGetToc(userId, nodeIds)

2. 收集所有目录中的节点ID
   Set<Long> allNodeIdsInTocs = ...

3. 一次性批量查询所有节点的完成状态
   completedNodes = domainService.getCompletedNodesInList(userId, allNodeIdsInTocs)

4. 遍历每个节点计算进度
```

**优势**：
- 减少数据库查询次数（N+1 问题优化）
- 适用于课程列表页、路线图页等批量场景

### 7.2 反向查找优化

**通过 nodes 字段快速定位**：

```sql
-- 传统方式：需要遍历所有课程，解析每个课程的 ToC，检查是否包含节点
-- 耗时：O(课程数 × ToC大小)

-- 优化方式：通过 nodes 字段直接查询
SELECT * FROM user_learning
WHERE user_id=?
AND object_type=2
AND nodes LIKE '%123%'

-- 耗时：O(1) - 索引查询
```

### 7.3 事件异步处理

**ToC 更新异步化**：
- 用户选择目录 → 立即返回
- `TocChosenEvent` 异步处理
- nodes 字段异步更新
- 不阻塞用户操作

## 十一、设计亮点总结

### 1. 双表分离设计
- **user_node_completion**：节点的绝对完成状态（跨课程、跨目录）
- **user_learning**：课程/目录的学习进度（基于当前目录动态计算）

### 2. 覆盖算法的优势
- 支持多种学习路径（细粒度/粗粒度）
- 避免强制完成所有子节点
- 数据不丢失（取消覆盖后立即恢复）

### 3. 事件驱动的自动化
- ToC 变化自动更新 nodes 字段
- 节点完成自动更新相关课程进度
- 异步处理，不阻塞主流程

### 4. 实时计算的准确性
- 进度基于当前状态实时计算
- 避免数据不一致
- state 字段动态计算，不存储冗余数据

### 5. 性能优化
- 批量查询减少 N+1 问题
- nodes 字段实现反向快速查找
- 分层架构清晰分离关注点
