# 前端代码整体优化规划

## 1. 现状分析

当前前端代码库遵循了 Vue 3 的基本最佳实践，包括使用 Composition API、Pinia 进行状态管理、Vue Router 进行路由控制等。项目结构清晰，具备一定的可维护性。

然而，随着业务功能的不断扩展，代码库规模持续增长，暴露出一些可优化的点：

-   **组件和视图结构扁平化**：`components` 和 `views` 目录下文件数量过多，导致查找和维护困难。
-   **服务层耦合度高**：`apiServiceV1.js` 文件趋向于“上帝对象”，包含了所有 API 调用，不利于按模块维护和扩展。
-   **路由管理集中化**：单一的路由配置文件在项目扩大后会变得臃肿。
-   **代码规范依赖手动执行**：缺少自动化的代码检查和格式化机制，难以保证长期一致性。
-   **性能优化空间**：部分页面和资源未实现懒加载，可能影响首屏加载速度。
-   **TypeScript 使用不充分**：部分逻辑文件仍为 `.js`，未能完全发挥 TypeScript 的类型优势。
-   **测试覆盖率低**：缺乏单元测试和集成测试，重构和功能迭代存在风险。

## 2. 优化目标

本次优化的核心目标是：

-   **提升可维护性**：通过模块化和代码规范，使代码更易于理解和修改。
-   **增强可扩展性**：建立清晰的架构，方便未来快速、安全地添加新功能。
-   **提高开发效率**：标准化开发流程，减少重复劳动，降低沟通成本。
-   **优化应用性能**：提升用户体验，特别是首屏加载速度。
-   **保障代码质量**：通过自动化工具和测试，确保代码的健壮性和稳定性。

## 3. 优化规划 (三阶段)

### 阶段一：结构整理与规范化 (低风险，高收益)

**目标**：整理现有代码结构，使其更加清晰、模块化，为后续优化打下基础。

1.  **组件与视图模块化**
    -   **`components` 目录**：按功能或业务模块创建子目录（如 `common`, `roadmap`, `course`, `user`），将现有组件归类。
    -   **`views` 目录**：按业务模块创建子目录（如 `admin`, `auth`, `course`, `user`），将页面组件归类，并同步更新路由导入路径。

2.  **API 服务层拆分**
    -   将 `services/api/v1/apiServiceV1.js` 拆分为多个按资源划分的服务文件（如 `systemService.js`, `userService.js`, `courseService.js` 等）。
    -   在 `services/api/v1/index.js` 中统一导出所有服务，方便统一管理和调用。

3.  **常量与工具函数整理**
    -   **常量**：在 `constants/` 目录下创建更具体的文件，如 `apiEndpoints.js`, `enums.js`。
    -   **工具函数**：审查 `utils/` 目录，确保只存放纯函数工具。

### 阶段二：性能优化与代码质量提升 (中风险，中收益)

**目标**：提升应用性能和代码健壮性。

1.  **路由懒加载**
    -   修改 `router/index.js`，对非核心或大型页面（如 `Admin.vue`, `Course.vue`）使用动态 `import()` 实现懒加载。

2.  **状态管理 (Pinia) 优化**
    -   审查现有 store，确保职责单一。为新的独立业务模块创建独立的 store。

3.  **代码规范与 Linting**
    -   引入更严格的 ESLint 和 Prettier 规则。
    -   配置 `lint-staged` 和 `husky`，在代码提交时自动执行格式化和检查。

### 阶段三：高级重构与现代化 (高风险，长期收益)

**目标**：引入更现代化的技术和模式，提升开发体验和长期可维护性。

1.  **TypeScript 全覆盖**
    -   将所有 `.js` 文件（特别是 `composables`, `services`, `stores`）重构为 `.ts`，并添加完整的类型定义。

2.  **测试覆盖**
    -   引入 Vitest 和 Vue Testing Library，为核心组件和业务逻辑编写单元测试和集成测试。

3.  **构建工具与依赖升级**
    -   定期审查并升级 Vite、Vue、Pinia 等核心依赖。
    -   分析构建产物，优化 chunk 分割策略。
